## 一、测试总结

第一轮 PRD 中列出的问题 **全部仍然存在**，本轮测试进一步确认了核心痛点并新增了两个维度的问题：

1. **进程勘探深度不足** — 点击进程后无法查看目录、关系、详情，深层信息完全不可达
2. **启动时不主动勘探** — 功能数据是"点击才加载"的惰性模式，应在启动时就完成系统级扫描
3. **窗口管理仍是核心痛点** — AI 窗口自命名、分组布局、监控进度三大功能依然不可用
4. **主题仅换色** — 视觉传达体系的差异化尚未实现
5. **视觉布局自适应缺失** — 界面元素不能随窗口大小自动调整，过于拥挤

## 二、新增/强化问题清单

### 2.0 [NEW] 启动时主动勘探 (P0)

**现状问题：** 应用启动后，进程/端口/窗口数据不会自动加载。用户需要手动点击对应 Tab（进程、端口、窗口）后才开始扫描。这意味着：
- 刚打开应用时看到的是空白或骨架屏
- 切换 Tab 时有明显的加载延迟
- 无法在后台持续跟踪系统状态变化

**需求：**
- [ ] 应用启动时 **立即启动后台扫描引擎**，并行扫描进程/端口/窗口
- [ ] 扫描结果缓存在内存中，切换 Tab 时 **即时渲染**，无需重新扫描
- [ ] 后台扫描定时刷新（进程 2s / 端口 5s / 窗口 3s），保持数据新鲜
- [ ] 首屏加载时显示带进度的初始化状态（"正在扫描进程... 328 个进程已发现"）
- [ ] 提供系统状态摘要面板（Dashboard），启动即可看到：已发现进程数、活跃端口数、窗口数、AI 工具实例数

### 2.1 [强化] 进程深层勘探不可达 (P0)

**上轮状态：** PRD 已列出需求（2.5/2.6），但本轮测试确认：点击进程后 **完全无法** 探查以下信息。

**细化需求：**

#### 2.1.1 进程目录探查
- [ ] 点击进程可查看其 **工作目录（CWD）**
- [ ] 提供"在资源管理器中打开"按钮
- [ ] 显示进程 **可执行文件完整路径**
- [ ] 如果是 Node.js/Python 等解释器进程，解析出实际运行的脚本路径

#### 2.1.2 进程关系探查
- [ ] 展示 **完整父子链**（从 root → 当前进程 → 所有子进程）
- [ ] 展示 **端口关联**（该进程监听/连接的所有端口）
- [ ] 展示 **文件锁关联**（可选，取决于 API 可达性）
- [ ] 关系数据 **以可折叠树 + 神经图双视图** 呈现

#### 2.1.3 进程详情面板
- [ ] 点击进程弹出 **详情侧边栏/抽屉**，包含：
  - 基础信息：PID、PPID、名称、路径、命令行参数、启动时间
  - 资源占用：CPU%（实时折线图）、内存（RSS/VMS）、线程数、句柄数
  - 网络连接：完整的 TCP/UDP 连接列表（本地地址:端口 → 远程地址:端口，状态）
  - 环境变量：完整的 Key=Value 列表（可搜索、可复制）
  - 模块/DLL：已加载模块列表（可选）
  - 操作按钮：结束进程、结束进程树、设置优先级、打开文件位置

### 2.2 [强化] 窗口管理三大功能 (P0 — 重中之重)

**上轮状态：** PRD 2.1/2.2/2.3 已列出，但本轮测试确认三项功能 **均不可用**。

#### 2.2.1 AI 窗口自命名（仍不可用）

**深层需求细化：**
- [ ] 检测到 AI CLI 窗口时 **自动分配默认名称**（格式：`{工具类型}-{序号}`，如 `Claude Code-1`）
- [ ] 用户可通过以下方式修改名称：
  - 双击窗口卡片标题 → 内联编辑
  - 右键菜单 → 重命名
  - 卡片上的编辑图标
- [ ] 名称 **持久化**：存储到 electron-store，匹配键为 `窗口标题前缀 + 可执行文件路径`
- [ ] 窗口关闭后名称保留，下次同路径窗口打开时 **自动恢复**
- [ ] AI 窗口在列表中 **置顶显示**，并有醒目的视觉标记（图标/边框颜色/标签）

#### 2.2.2 通知携带窗口名称（仍不可用）

**通知体验要求：**
- [ ] 任务完成通知格式：`[{自定义名称}] 任务完成`
  - 例：`[Claude-前端重构] 任务完成`
  - 若无自定义名称：`[Claude Code (PID:12345)] 任务完成`
- [ ] 通知正文包含：工具类型、PID、任务持续时间
- [ ] 点击通知 → 直接聚焦到对应窗口（调用 focusWindow）
- [ ] 通知要使用系统原生通知 API（Windows Toast Notification），而非仅应用内 toast

#### 2.2.3 分组/布局功能（仍不可用）

**排查方向：**
- [ ] 确认 IPC 通道 `window:createGroup` / `window:saveLayout` / `window:restoreLayout` 是否已注册
- [ ] 确认 PowerShell/Win32 API 调用链是否有 C# 编译问题（类似 focusWindow 的 C#5 兼容性问题）
- [ ] 确认 UI 按钮事件绑定是否正确
- [ ] 修复后需验证：创建分组 → 分组内窗口操作（最小化/恢复/排列）→ 删除分组 全链路
- [ ] 修复后需验证：保存布局 → 手动打乱窗口位置 → 恢复布局 全链路

#### 2.2.4 监控进度（仍不够好）

**进一步细化：**
- [ ] 当前进度检测颗粒度太粗，只有"活跃/空闲"两态
- [ ] 需要的颗粒度：
  - `idle` — 无活动（终端空闲，无输出）
  - `thinking` — AI 模型在思考（CPU 高，无终端输出）
  - `coding` — AI 正在写代码（有持续终端输出 + 文件写入）
  - `waiting-input` — 等待用户输入（终端提示符出现，CPU 平稳）
  - `compiling` — 在运行编译/测试（检测 npm/tsc/vite 等命令执行）
  - `completed` — 任务完成（关键词检测：Done/Complete/✓/finished）
  - `error` — 检测到错误（关键词：Error/Failed/✗/panic）
- [ ] 每次状态变化记录时间戳，构建 **进度时间线**
- [ ] 时间线可在 UI 中以时序图形式展示

### 2.3 [强化] 主题视觉传达 (P2)

**上轮状态：** PRD 2.8 已列出，本轮测试再次确认仅换色。

**核心强调：** 主题不是调色板，是一套完整的设计语言。每个主题应在以下 **全部** 维度产生差异：

| 维度 | Soviet Constructivism | Cyberpunk Neon | Swiss Minimal |
|------|----------------------|----------------|---------------|
| 组件排布 | 紧凑、高密度、不对称 | 适中密度、发光间距 | 宽松、大量留白 |
| 字体 | Mono 刚健、大写标题 | 未来科技体、发光 | Helvetica 经典无衬线 |
| 圆角 | 0-2px 方角 | 4-8px + 发光边缘 | 0px 纯方角 |
| 阴影 | 无阴影，用边框 | 发光阴影（glow） | 无/极淡阴影 |
| 动效 | 刚性、快速、无弹跳 | 流畅、全息闪烁 | 极简、仅功能性过渡 |
| 装饰 | 对角线条纹、几何切割 | 网格背景、粒子、扫描线 | 无装饰、纯网格对齐 |
| 信息密度 | 高 | 中 | 低 |
| 卡片风格 | 边框突出、小间距 | 发光边缘、适中间距 | 极简线框、大间距 |

### 2.4 [NEW] 视觉布局自适应 (P1)

**现状问题：** 界面元素在不同窗口尺寸下表现不佳：
- 窗口缩小时组件挤压变形，文字截断
- 窗口放大时组件未充分利用空间
- 没有响应式断点系统
- 文字大小、间距、卡片列数无法根据容器尺寸自动调整

**需求：**
- [ ] 引入 **CSS Container Queries** 实现组件级响应式
- [ ] 定义标准断点（xs/sm/md/lg/xl），每个断点对应不同的：
  - 列数（1/2/3/4 列网格）
  - 文字大小（阶梯缩放）
  - 间距/内边距
  - 信息密度（紧凑/标准/宽松）
- [ ] 数值格式随空间变化（完整数字 → 缩写 → 图标+数字）
- [ ] 侧边栏可折叠/展开，释放更多空间给主内容区
- [ ] 所有面板区域支持 **可拖拽调整大小**
