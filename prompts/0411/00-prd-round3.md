## 二、问题清单（本轮新增 + 持续存在）

### 3.1 [NEW/P0-Blocker] 启动黑屏 — 缺少加载页

**现象**：应用启动后窗口一片纯黑，持续数秒后内容才可能出现（或始终黑屏）。用户无法判断应用是在加载还是已崩溃。

**根因分析**：
- `BackgroundScannerManager` 启动后所有扫描器立即被 `stopped`（日志可见）
- `scannerInitStatus` 停留在 `'loading'`，`InitializationScreen` 组件渲染但背景色 `bg-surface-950` 与纯黑几乎无差异
- 可能的情况：renderer 进程 JS 报错导致 React 未挂载，或 CSS 变量未及时加载
- Electron `BrowserWindow` 设置了 `show: false` + `ready-to-show` 才显示，但 Vite dev server 的 HMR 连接可能存在时序问题

**需求**：
- [ ] **实现原生闪屏（Splash Screen）**：在 Electron 主进程层面，使用 `BrowserWindow` 创建一个轻量闪屏窗口（纯 HTML，不依赖 React/Vite），启动时立即显示
- [ ] 闪屏内容：应用 Logo + "DevHub" 名称 + 加载进度条 + 当前加载阶段文字
- [ ] 加载阶段：`正在初始化...` → `正在启动扫描引擎...` → `正在扫描系统...` → `准备就绪`
- [ ] 主窗口 `ready-to-show` 后，闪屏淡出消失，主窗口淡入显示
- [ ] **备选方案**：如果不用独立闪屏窗口，则在 `index.html` 的 `<body>` 中直接用纯 CSS/HTML 写一个 loading 占位（不依赖 React hydration），React 挂载完成后替换
- [ ] 确保在 dev 模式（Vite HMR）和 production 模式下都不会出现黑屏

### 3.2 [强化/P1] UI 显示过于局促

**现象**：界面元素紧密排列，文字被截断，卡片间距不足，整体观感拥挤。（截图已记录）

**与前轮关系**：Round 2 的 `2.4 视觉布局自适应` 已提出，但问题依然存在。

**需求细化**：
- [ ] **增大卡片间距**：当前卡片间 `gap` 过小，至少增加到 `12px`（当前估计 `4-8px`）
- [ ] **增大内边距**：面板内容区 padding 从当前的紧凑值增加到至少 `16px`
- [ ] **文字截断处理**：长文本使用 `tooltip` 而非 `overflow: hidden`，确保用户能看到完整信息
- [ ] **进程/端口卡片**：宽度不足时不应挤压内容，而是减少列数
- [ ] **引入响应式断点系统**：
  - `< 1000px`：1列布局，侧边栏折叠
  - `1000px - 1400px`：2列，标准间距
  - `1400px - 1800px`：3列，标准间距
  - `> 1800px`：4列，宽松间距
- [ ] **信息密度可调**：提供"紧凑/标准/宽松"三级密度切换（存入设置持久化）

### 3.3 [持续/P0-Critical] 窗口管理核心功能全面失效

**这是三轮测试中反复提出的第一优先级问题。**

#### 3.3.1 AI 窗口自命名（三轮未修复）

**问题**：用户同时运行多个 AI CLI 工具实例（Claude Code × N、Codex CLI、OpenCode、Gemini CLI 等），无法区分各实例。

**完整需求**：
- [ ] 检测到 AI CLI 窗口时 **自动分配默认名称**：`{工具名}-{序号}`（如 `Claude Code-1`、`Codex CLI-2`）
- [ ] 用户可自定义命名（双击标题内联编辑 / 右键菜单重命名 / 卡片编辑图标）
- [ ] 名称持久化到 `electron-store`，匹配键为 `窗口标题前缀 + 可执行文件路径 + PID`
- [ ] 窗口关闭后名称保留，相同路径窗口再次打开时自动恢复
- [ ] AI 窗口在列表中 **置顶 + 视觉高亮**（专属图标/彩色边框/标签徽章）
- [ ] 搜索框支持按自定义名称搜索

#### 3.3.2 任务完成通知必须携带窗口名称（三轮未修复）

**问题**：任务完成通知无法区分来源，多实例场景下完全无用。

**完整需求**：
- [ ] 通知标题格式：`[{自定义名称}] 任务完成`
  - 有自定义名称：`[Claude-前端重构] 任务完成`
  - 无自定义名称：`[Claude Code (PID:12345)] 任务完成`
- [ ] 通知正文：工具类型 + PID + 任务持续时间 + 最后输出摘要（最后 3 行终端输出）
- [ ] 使用 **Windows 原生 Toast Notification**（不仅是应用内 toast）
- [ ] 点击通知 → 聚焦对应窗口（需 focusWindow 可用）
- [ ] 通知历史记录面板（最近 50 条，可滚动查看）

#### 3.3.3 AI 任务完成感测严重不准（三轮持续存在）

**问题**：AI 工具任务完成检测经常误报、瞎报、错报：
- **误报**：AI 还在工作，通知说"任务完成"
- **漏报**：AI 已完成，但迟迟没通知
- **错报**：检测到错误关键词就报"完成"，实际是报错

**根因推断**：当前检测逻辑过度依赖简单关键词匹配，未结合上下文和进程状态。

**需求**：
- [ ] **多信号融合检测**，不能仅靠关键词：
  1. **终端输出关键词**：仅作为候选信号，需配合其他信号确认
  2. **CPU 活动**：AI 模型运行时 CPU 明显偏高，完成后回落
  3. **终端输出速率**：编码阶段输出密集，完成后输出停止或仅有提示符
  4. **输入提示符检测**：终端回到 `$`/`>`/`❯` 等提示符 = 任务完成或等待输入
  5. **进程状态变化**：子进程退出、线程数变化
- [ ] **置信度评分**：综合以上信号输出 0-100 置信度，仅在 ≥ 80 时发送通知
- [ ] **确认窗口期**：检测到"可能完成"后，等待 3-5 秒确认无新输出再通知
- [ ] **用户标记纠错**：通知中提供"标记为误报"按钮，用于自动调整检测模型
- [ ] **各 AI 工具独立配置**：不同 CLI 工具的输出格式不同，需分别配置检测规则

#### 3.3.4 分组和布局功能不可用（三轮未修复）

**问题**："分组"和"布局"按钮存在，但点击后无反应或行为异常。

**完整需求**：
- [ ] **排查并修复分组功能**：
  - 确认 IPC 通道 `window:createGroup` 等是否已注册
  - 确认 PowerShell/Win32 API 调用链中的 C# 编译问题（同 focusWindow 的 C#5 兼容性）
  - 修复后验证：创建分组 → 添加/移除窗口 → 分组内批量操作（最小化/恢复/排列）→ 删除分组
- [ ] **排查并修复布局功能**：
  - 保存当前窗口布局 → 手动打乱窗口位置 → 一键恢复布局
  - 支持多套布局（"开发"、"演示"、"休息"等场景）
  - 布局持久化到 `electron-store`
- [ ] **新增布局预设**：
  - 平铺（Tile）：等分屏幕给所有窗口
  - 主次分区（Master-Slave）：主窗口占大面积，其余小排列
  - 层叠（Cascade）：窗口斜向层叠排列

#### 3.3.5 监控进度功能不够好（三轮持续存在）

**问题**：进度状态粗糙，只有"活跃/空闲"两态。

**完整需求**：
- [ ] 实现精细状态机（7 种状态）：
  ```
  idle → thinking → coding → compiling → waiting-input → completed → error
  ```
- [ ] 每种状态有独立的视觉表示（图标 + 颜色 + 动画）
- [ ] 状态变化记录时间戳，构建进度时间线
- [ ] 时间线 UI：水平时序条，鼠标悬停显示详情
- [ ] 支持自定义状态检测规则（用户可为特定工具配置）

#### 3.3.6 窗口功能过于有限（新增细化）

**问题**：窗口管理的可操作功能太少。

**需求**：
- [ ] **基础操作**：聚焦 / 最小化 / 最大化 / 还原 / 关闭 / 置顶(Always on Top)
- [ ] **高级操作**：
  - 发送按键到窗口（如 Ctrl+C 终止命令）
  - 窗口截图
  - 窗口透明度调节
  - 窗口移动到指定显示器
- [ ] **批量操作**：
  - 全部最小化 / 全部还原
  - 选中多个窗口 → 批量操作
  - 按分组批量操作
- [ ] **快捷键**：
  - 全局热键切换到指定 AI 窗口（可自定义）
  - 循环切换 AI 窗口

### 3.4 [NEW/P0] 端口探查极慢

**现象**：点击端口详情后，右侧面板显示 "Loading port data..." 持续很长时间（截图已记录），用户等待体验极差。

**根因推断**：
- 每次点击端口详情都重新执行 `netstat` 或 PowerShell 查询
- 未利用 `BackgroundScannerManager` 已有的缓存数据
- 可能存在不必要的全量扫描（查询单个端口的信息不需要扫描全部端口）

**需求**：
- [ ] **利用缓存优先**：端口列表数据已在后台扫描器中，详情面板应首先从缓存读取
- [ ] **增量查询**：点击端口详情时，仅查询该端口的增量信息（关联进程详情、远程连接等），而非全量重扫
- [ ] **加载骨架屏**：查询期间显示骨架屏（skeleton），而非空白 + 旋转图标
- [ ] **超时机制**：查询超过 3 秒自动降级为缓存数据 + "部分信息可能不是最新" 提示
- [ ] **并行查询**：多个端口的详情查询应并行执行，不阻塞 UI
- [ ] **取消查询**：用户切换到其他端口时，自动取消前一个端口的查询

### 3.5 [持续/P2] 主题系统 — 不是换颜色

**现象**：切换主题后仅颜色变化，组件排布、字体、圆角、阴影、动效、信息密度等维度无差异。

**三轮持续强调**，核心原则：**主题 = 完整的视觉传达体系，不是调色板。**

**需求**（与 R1/R2 一致，增加执行标准）：

| 维度 | Soviet Constructivism（当前） | Cyberpunk Neon | Swiss Minimal |
|------|-------------------------------|----------------|---------------|
| **组件排布** | 紧凑、高密度、不对称布局 | 适中密度、发光间距 | 宽松、大量留白、网格对齐 |
| **字体** | Mono 刚健、大写标题、紧凑字距 | 未来科技体、发光文字效果 | Helvetica/Inter、经典无衬线 |
| **圆角** | 0-2px 方角 | 4-8px + 发光边缘 | 0px 纯方角 |
| **阴影** | 无阴影，用粗边框 | 发光阴影（neon glow） | 无/极淡投影 |
| **动效** | 刚性、快速、无弹跳 | 流畅、全息闪烁、扫描线 | 极简、仅功能性过渡 |
| **装饰** | 对角线条纹、几何切割、星章 | 网格背景、粒子、数据流 | 无装饰、纯网格对齐 |
| **信息密度** | 高 | 中 | 低 |
| **卡片** | 粗边框、小间距、直角 | 发光边缘、悬浮感 | 极简线框、大间距 |
| **按钮** | 方角、粗实心、大写标签 | 发光轮廓、悬浮 | 极简线框、小号 |

**执行标准**：
- [ ] 每个主题作为独立的 CSS token 集合 + Tailwind 变体类
- [ ] 主题切换需同时改变：颜色变量、间距变量、圆角变量、字体变量、动效变量
- [ ] 组件需根据主题变量自适应（如 `border-radius: var(--radius-card)` 而非硬编码）
- [ ] 至少 3 套有本质差异的完整主题

### 3.6 [NEW/P0-Critical] 性能与安全深度问题

**这是本轮新增的重要维度。**

#### 3.6.1 性能问题

**现象与推断**：
- 端口探查慢（3.4 已详述）
- 扫描器启动后立即 stopped（可能因为异常退出未处理）
- 前端渲染大量进程/端口数据时可能存在性能瓶颈

**需求**：
- [ ] **渲染优化**：
  - 进程/端口/窗口列表使用 **虚拟滚动**（已引入 `@tanstack/react-virtual`，确认是否正确应用）
  - 避免不必要的 re-render（检查 zustand store 的 selector 粒度）
  - 大数据量场景（500+ 进程、100+ 端口）的渲染帧率测试
- [ ] **后台扫描优化**：
  - 扫描器异常时自动重启（当前 stopped 后不恢复）
  - 扫描间隔动态调节：窗口聚焦时用高频（进程 2s/端口 5s），最小化时降频（10s/30s）
  - 扫描结果差分传输，避免全量推送
- [ ] **内存管理**：
  - 监控 Electron 主进程和渲染进程内存占用
  - 设置进程数据保留上限（如最多缓存 2000 个进程记录）
  - 定期清理过期的扫描缓存
- [ ] **IPC 性能**：
  - 避免高频 IPC 调用（合并批量数据推送）
  - 使用 `structured clone` 传输大对象
  - 限制单次 IPC 传输大小

#### 3.6.2 安全问题

**需求**：
- [ ] **CSP (Content Security Policy)**：
  - 配置严格的 CSP 头：禁止内联脚本、限制资源来源
  - `script-src 'self'`（生产）/ `script-src 'self' http://localhost:*`（开发）
  - `style-src 'self' 'unsafe-inline'`（Tailwind 需要内联样式）
- [ ] **Preload 脚本安全**：
  - 确认 `contextIsolation: true` 和 `sandbox: true` 已启用（已确认）
  - 确认 `nodeIntegration: false`（已确认）
  - 审查所有 `contextBridge.exposeInMainWorld` 暴露的 API 是否有越权风险
- [ ] **IPC 参数校验**：
  - 所有 IPC handler 的输入参数必须做 schema 校验
  - 禁止直接将用户输入传入 `child_process.exec` 等敏感 API
  - PowerShell 命令注入防护：所有参数需转义
- [ ] **进程操作安全**：
  - 结束进程 / 结束进程树操作需二次确认
  - 系统关键进程（PID < 100、System、csrss 等）禁止操作
  - 操作日志记录（谁在什么时间结束了哪个进程）
- [ ] **数据存储安全**：
  - `electron-store` 数据文件的权限检查
  - 不存储敏感信息（如进程的环境变量中可能包含密钥/token）
  - 清理缓存时确保彻底删除
- [ ] **更新安全**：
  - 如有自动更新，确认使用签名验证
  - 检查依赖项是否有已知漏洞（`pnpm audit`）
