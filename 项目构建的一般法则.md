# 项目构建的一般法则

> AI 驱动开发的系统性回环：从模糊想法到可交付产品的完整路径

---

## Phase 1: 确定需求

**目标**：将模糊的想法转化为清晰、可执行的需求边界

**核心活动**：
- 与 AI 进行多轮对话，澄清「我到底想要什么」
- 区分「必须有」与「最好有」
- 识别核心用户场景（谁在什么情况下用这个做什么）
- 定义成功标准（怎样算做完了）

**输出物**：
- 一句话描述：这个项目是什么
- 核心场景列表（3-5 个）
- 明确的边界：这个项目「不做」什么

**质量检查**：
- [ ] 能否用一句话向外行解释清楚？
- [ ] 核心场景是否覆盖 80% 的使用情况？
- [ ] 边界是否足够清晰，不会无限蔓延？

---

## Phase 2: 撰写文档

**目标**：将需求固化为可追溯的文字契约

**核心活动**：
- 编写产品需求文档（PRD）
- 定义功能清单与优先级
- 明确技术约束与依赖
- 预判风险点

**输出物**：
- `docs/PRD.md` — 产品需求文档
- `docs/TECH_SPEC.md` — 技术规格说明
- `docs/RISK.md` — 风险预判与应对

**质量检查**：
- [ ] 文档是否足够让 AI 理解并独立执行？
- [ ] 是否有遗漏的边界情况？
- [ ] 技术选型是否合理？

---

## Phase 3: UML 工程化

**目标**：将脑中的思绪用程序员的语言规范化表达

**核心理念**：
> 在写一行代码之前，先用图把系统画清楚。
> UML 是你与 AI 之间的「工程共识语言」。

**图类型与用途**：

| 图类型 | 回答的问题 | 适用阶段 |
|--------|-----------|---------|
| **用例图 (Use Case)** | 系统能做什么？谁来用？ | 需求确认 |
| **类图 (Class)** | 数据长什么样？有哪些实体？ | 数据建模 |
| **时序图 (Sequence)** | 组件之间怎么交互？ | 接口设计 |
| **活动图 (Activity)** | 业务流程怎么走？ | 流程梳理 |
| **状态图 (State)** | 对象有哪些状态？如何转换？ | 状态机设计 |
| **组件图 (Component)** | 系统分几个模块？怎么组织？ | 架构划分 |
| **部署图 (Deployment)** | 怎么部署？跑在哪里？ | 运维规划 |

**核心活动**：
1. 用例图：画出所有角色与系统功能的边界
2. 类图：定义核心数据模型与关系
3. 时序图：描绘关键流程的调用链
4. 活动图：梳理复杂业务逻辑的分支
5. 状态图：明确有状态对象的生命周期
6. 组件图：划分模块边界与依赖关系

**输出物**：
- `docs/uml/use-case.puml` — 用例图
- `docs/uml/class.puml` — 类图
- `docs/uml/sequence-*.puml` — 关键流程时序图
- `docs/uml/activity-*.puml` — 业务活动图
- `docs/uml/state-*.puml` — 状态图
- `docs/uml/component.puml` — 组件图

**质量检查**：
- [ ] 用例图是否覆盖所有核心场景？
- [ ] 类图是否反映真实的数据关系？
- [ ] 时序图能否指导 API 设计？
- [ ] AI 能否仅凭这些图理解系统全貌？

---

## Phase 4: 搭建原型

**目标**：用最小成本验证核心假设

**核心活动**：
- 构建可交互的 UI 原型（低保真/高保真）
- 验证核心流程的可行性
- 收集早期反馈
- 快速迭代调整

**输出物**：
- 可点击的原型（Figma / 代码原型）
- 用户反馈记录
- 调整后的需求修订

**质量检查**：
- [ ] 核心流程是否跑通？
- [ ] 用户是否理解如何使用？
- [ ] 是否发现需求遗漏？

---

## Phase 5: 约定架构

**目标**：在动手之前，确定技术骨架

**核心活动**：
- 确定技术栈选型
- 定义目录结构规范
- 约定代码风格与命名规则
- 设计 API 接口契约
- 规划数据库 schema

**输出物**：
- `docs/ARCHITECTURE.md` — 架构设计文档
- `docs/API.md` — API 接口定义
- `docs/DATABASE.md` — 数据库设计
- 项目脚手架代码

**质量检查**：
- [ ] 架构是否支持未来扩展？
- [ ] 是否有明确的模块边界？
- [ ] AI 能否按此架构独立开发？

---

## Phase 6: 开发构建

**目标**：将设计转化为可运行的代码

**核心活动**：
- 按模块分解任务（TodoList 驱动）
- TDD 驱动开发：Red → Green → Refactor
- 持续集成：每个功能完成即测试
- 代码审查：AI 自审 + 人工抽查

**开发原则**：
- 一次只做一件事
- 每个提交都应该是可运行的
- 不留 TODO，不埋坑
- 代码即文档

**输出物**：
- 可运行的代码
- 单元测试覆盖
- 提交历史记录

**质量检查**：
- [ ] 测试是否覆盖核心路径？
- [ ] 代码是否符合约定规范？
- [ ] 是否有明显的性能问题？

---

## Phase 7: 批判性检测

**目标**：以攻击者视角审视系统

**核心活动**：
- 功能测试：所有功能是否按预期工作
- 边界测试：异常输入、极端情况
- 性能测试：压力、负载、响应时间
- 安全审计：注入、越权、泄露
- 用户体验走查：真实用户视角

**输出物**：
- 测试报告
- Bug 列表与修复记录
- 性能基准数据

**质量检查**：
- [ ] 所有已知 Bug 是否已修复？
- [ ] 是否通过安全扫描？
- [ ] 性能是否达标？

---

## Phase 8: 权限/身份/安全/功能

**目标**：确保系统的安全性与合规性

**核心活动**：
- 身份认证机制实现
- 权限控制矩阵设计
- 数据加密与脱敏
- 审计日志记录
- 合规性检查

**输出物**：
- 权限矩阵文档
- 安全配置清单
- 审计日志方案

**质量检查**：
- [ ] 是否有越权漏洞？
- [ ] 敏感数据是否加密？
- [ ] 是否满足合规要求？

---

## Phase 9: 部署交付

**目标**：将产品安全地推向用户

**核心活动**：
- 环境配置（开发/测试/生产）
- CI/CD 流水线搭建
- 灰度发布策略
- 回滚预案准备
- 上线检查清单

**输出物**：
- 部署脚本与配置
- 发布检查清单
- 回滚操作手册

**质量检查**：
- [ ] 是否支持一键部署？
- [ ] 是否有回滚能力？
- [ ] 监控告警是否就位？

---

## Phase 10: 监控运维

**目标**：保障系统持续健康运行

**核心活动**：
- 日志收集与分析
- 性能监控与告警
- 错误追踪与响应
- 用户行为分析
- 定期健康检查

**输出物**：
- 监控仪表盘
- 告警规则配置
- 运维手册

**质量检查**：
- [ ] 是否能快速定位问题？
- [ ] 告警是否及时有效？
- [ ] 是否有容量规划？

---

## Phase 11: 反馈迭代 ↺

**目标**：形成持续改进的闭环

**核心活动**：
- 收集用户反馈
- 分析使用数据
- 识别改进机会
- 规划下一版本
- **回到 Phase 1，开启新循环**

**输出物**：
- 用户反馈汇总
- 迭代计划
- 版本路线图

**质量检查**：
- [ ] 是否有系统性的反馈收集机制？
- [ ] 改进是否基于数据驱动？
- [ ] 是否形成持续迭代的节奏？

## 使用指南

### 与 AI 协作的关键原则

1. **每个 Phase 都要有明确的输出物**
   - 输出物是下一阶段的输入
   - 输出物是与 AI 沟通的「上下文锚点」

2. **质量检查是阶段门禁**
   - 不通过则不进入下一阶段
   - 宁可回退也不带伤前进

3. **UML 图是核心沟通语言**
   - AI 读图比读长文更精准
   - 图是需求变更的唯一来源

4. **回环是常态，不是例外**
   - 预期会多次迭代
   - 每次迭代都留下文档痕迹

### 快速检查清单

在任何阶段感到迷茫时，问自己：
- [ ] 当前阶段的目标是什么？
- [ ] 我需要产出什么？
- [ ] 质量标准是什么？
- [ ] 下一阶段需要什么输入？
